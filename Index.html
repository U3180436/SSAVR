Functional Requirements

This webpage discusses functional requirements in software architecture. It explains that functional requirements are product features that must be implemented to enable users to achieve their goals. The page notes that functional requirements are different from other types of requirements and are crucial elements of Agile Development. It further explains that functional requirements are written as a part of a user story, which is a key component of Agile Development. The webpage provides an example of a user story and the functional requirements associated with it.

Quality Attribute Requirements

This webpage discusses Quality Attribute Requirements (QAR) or Non-Functional Requirements, which are used to validate Functional Requirements in Agile Development. QAR provides context behind different aspects of each functional requirement, and there are several main quality attributes such as availability, interoperability, modifiability, performance, security, testability, usability, and scalability. QAR can relate to Agile Development through the different requirements that need to be met to achieve a functional requirement.

Constraints

This webpage explains what constraints are in software architecture, which are fixed design restrictions put in place by stakeholders and users. They are critical to software development, as they ensure that the architecture is built on top of the listed constraints and cannot be changed without breaking the necessary requirements. Constraints can be technical or business-oriented and are established by stakeholders and project delivery leads. Technical constraints relate to design decisions that are architectural drivers, while business constraints relate to decisions that restrict software architecture design due to the resources available to the development team.

Quality Attribute Scenarios

This webpage discusses the importance of measuring software quality and how it is affected by software architecture. Software quality attributes describe a system's intended behavior within the environment for which it was built and provide a means of measuring the fitness and suitability of a product. Quality attributes must be described in terms of scenarios to provide quantifiable arguments about a system. Scenarios help describe the qualities of a system, but architectural tactics describe how a given quality can be achieved. The webpage emphasizes the importance of identifying desired system qualities before a system is built to select or create the right architecture.

SOFTWARE SYSTEMS ARCHITECTURE

On this website, you will learn how to produce software architecture design from system requirements; This website covers various concepts and techniques used in the design of software architectures, including architectural requirements, architectural styles, architecture documentation and architecture evaluation. You will learn about the role of architecture and architects in software engineering. Modern architectures such as service-oriented and cloud architecture and wireless systems architecture are also covered in detail.

Topics

What you can expect to learn by exploring this website will be:

  • Develop an understanding of the principles of software architecture, architectural best-practices, and how architecture is used in modern software engineering;
  • Understand the role of a software architect in software engineering practice;
  • Examine and compare various architecture styles and solutions;
  • Design an architecture that reflects and balances the different needs of its stakeholders; and
  • Communicate the architecture to stakeholders and demonstrate that it has met their requirements.
