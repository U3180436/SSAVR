Quality Attribute Requirements

What are Quality Attribute Requirements in Software Architecture

Quality Attribute Requirements (QAR), a.k.a. Non-Functional Requirements, are a means of validating the Functional Requirements created in Agile Development. Each QAR provides a more in-depth context behind different aspects of each functional Requirement (Carneige Mellon University 2023).

For example, you may have a Functional Requirement where the "User must be able to login with their account credentials", a relavent qualification using a QAR may be how fast it takes for the system to log the user in. This example here is a referemce to the QAR of Performance .

How many Quality Attributes are there?

There is a large list of Quality Attributes, but the main ones that are considered are:

  • Availability - How available is a Web application to the user when it is updating?
  • Interoperability - How does the system retrieve data from other systems?
  • Modifiability - How easily can developers modify the data?
  • Performance - How fast does the system respond when the user inputs data?
  • Security - How does the system handle unauthorized users?
  • Tetability - How easy is it for Testers to check the fucntionality of the sytem?
  • Usability - How easy is it for users to interact with the system?
  • Scalability - How well does the system accomodate a large number of users and data?

How do these Quality Attributes relate to Agile Development?

A Functional Requirement can have one or many Quality Attribute Requirements it must meet. For example, if a the Functional Requirement is "User must be able to login with their account credentials", then the QAR of Availability may be that "The system MUST be avalable to the user 24/7 so that they can login at any time."
